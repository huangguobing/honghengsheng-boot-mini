# 订单成本填充与应收应付业务重构设计

**日期**：2026-03-07
**状态**：已确认
**范围**：订单状态流转、采购信息填充、应收应付计划管理、结算闭环

---

## 一、业务背景

现有系统存在两个不闭环的问题：

1. **成本填充 = 状态推进，耦合过紧**：`fillOrderCost` 同时做了填采购信息 + 推状态到已完成 + 创建供应商付款单三件事
2. **应收无账期管理**：`markOrderAsPaid` 一步标记全额收款，没有和 PaymentPlan 打通

核心业务模型：**一个订单 = 一个客户 + 一个工地项目 + N个供应商采购（多采一销）**

---

## 二、订单状态流转重构

### 2.1 新状态枚举

| 值 | 名称 | 说明 |
|----|------|------|
| 0 | DRAFT（草稿） | 可编辑所有信息 |
| 1 | CONFIRMED（已确认） | 原 PENDING_LOAD，可填充采购信息、可发货 |
| 2 | SHIPPED（已发货） | 可填充/编辑采购信息、建立应收应付计划 |
| 4 | SETTLING（结算中） | 等待收付款，可逐笔核销 |
| 3 | COMPLETED（已完成） | 应收应付全部结清 |
| 50 | CANCELLED（已取消） | 终止 |

### 2.2 状态转换规则

```
DRAFT(0) ──提交──→ CONFIRMED(1)
DRAFT(0) ──取消──→ CANCELLED(50)

CONFIRMED(1) ──发货──→ SHIPPED(2)
CONFIRMED(1) ──取消──→ CANCELLED(50)

SHIPPED(2) ──进入结算──→ SETTLING(4)   [前置校验]
SHIPPED(2) ──取消──→ CANCELLED(50)

SETTLING(4) ──全部结清──→ COMPLETED(3) [自动判定 或 手动标记]
```

### 2.3 各状态可执行操作

| 状态 | 可执行操作 |
|------|-----------|
| 草稿 | 编辑订单、提交、取消 |
| 已确认 | 填充采购信息、发货、取消 |
| 已发货 | 填充/编辑采购信息、建立应收计划、建立应付计划、进入结算、取消 |
| 结算中 | 核销应收（登记收款）、核销应付（登记付款）、编辑计划 |
| 已完成 | 查看、编辑成本、退换货 |

## 三、采购信息填充重构

### 3.1 核心变更：解耦

`fillOrderCost` 只做一件事 — 填写采购信息，不再推状态，不再创建付款单。

### 3.2 填充内容（每个订单明细）

- 供应商ID
- 采购数量、采购单价、采购金额
- 税额
- 其他费用（订单级别）

### 3.3 移除字段

- `OrderItemDO.paymentDate` — 废弃/移除，付款信息由 PaymentPlan 独立管理
- `OrderItemDO.isPaid` — 废弃/移除
- `OrderCostFillReqVO.items[].paymentDate` — 移除
- `OrderCostFillReqVO.items[].isPaid` — 移除

### 3.4 自动计算（不变）

- 毛利 = 销售金额 - 采购金额
- 净利 = 毛利 - 税额 - 其他费用
- 汇总到订单级别

### 3.5 可操作状态

- 已确认、已发货状态下可自由填充和编辑
- 结算中状态下编辑需同步调整已有应付计划金额

## 四、应收应付计划统一管理

### 4.1 复用 PaymentPlan 表

现有 `erp_payment_plan` 表已有 `type` 字段（0=应付，1=应收），数据模型无需新建表。

### 4.2 应付计划（type=0，付供应商）

**创建时机**：采购信息填充后，已发货状态下手动创建

**操作流程**：
1. 管理员看到按供应商分组的采购汇总
2. 为每个供应商创建应付计划（可多期）
3. 每期手动填写：金额、计划付款日期、付款方式
4. 系统校验：各期金额之和 = 该供应商采购总额

### 4.3 应收计划（type=1，收客户款）

**创建时机**：已发货后手动创建

**操作流程**：
1. 管理员看到订单销售总额
2. 为客户创建应收计划（可多期）
3. 每期手动填写：金额、计划收款日期、收款方式
4. 系统校验：各期金额之和 = 订单应收总额

### 4.4 PaymentPlan 字段使用

| 字段 | 应付(type=0) | 应收(type=1) |
|------|-------------|-------------|
| type | 0 | 1 |
| orderId | 订单ID | 订单ID |
| supplierId | 供应商ID | null |
| customerId | null | 客户ID |
| stage | 第N期 | 第N期 |
| planAmount | 计划付款金额 | 计划收款金额 |
| planDate | 计划付款日期 | 计划收款日期 |
| actualAmount | 实际付款金额 | 实际收款金额 |
| actualDate | 实际付款日期 | 实际收款日期 |
| status | 0待付/10已付/20逾期/30取消 | 同左 |
| paymentMethod | 付款方式 | 收款方式 |

### 4.5 新增/调整 API

| API | 方法 | 说明 |
|-----|------|------|
| `/erp/payment-plan/create-payable` | POST | 为订单某供应商创建应付计划 |
| `/erp/payment-plan/create-receivable` | POST | 为订单创建应收计划 |
| `/erp/payment-plan/verify/{id}` | PUT | 核销（登记实际收/付款） |
| `/erp/payment-plan/list-by-order` | GET | 查询订单所有应收应付计划 |

### 4.6 Payment 表保留

Payment 表保留作为每个供应商的付款单汇总记录，与 PaymentPlan 的关系不变。

## 五、结算与完成判定

### 5.1 进入结算（手动触发）

**操作**：管理员在已发货状态下点击「进入结算」

**前置校验**：
- 所有订单明细的采购信息已填充
- 至少有一条应收计划
- 至少有一条应付计划
- 应收计划总额 = 订单应收总额
- 应付计划按供应商汇总 = 各供应商采购总额

**结果**：SHIPPED(2) → SETTLING(4)

### 5.2 完成判定

**自动完成**：每次核销后检查：
- 所有应收计划 status = 已付款(10)
- 所有应付计划 status = 已付款(10)
- 全部结清 → SETTLING(4) → COMPLETED(3)

**手动完成**（兜底）：管理员可手动标记完成，适用于小额尾差等情况

### 5.3 移除的功能

- `markOrderAsPaid()` — 被应收计划逐笔核销替代
- `fillOrderCost` 中的自动推状态逻辑 — 移除
- `fillOrderCost` 中的自动创建付款单逻辑 — 移除
- `validateSupplierPaymentConsistency()` — 移除（不再需要）
